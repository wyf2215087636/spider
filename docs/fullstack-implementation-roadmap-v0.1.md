# AI研发协作平台 - 全栈逐步落地路线图 v0.1

- 版本：v0.1
- 日期：2026-03-01
- 适用阶段：从0到MVP首版上线
- 决策状态：执行草案

## 1. 目标

在同一个工作空间内并行推进前端与后端，按“契约先行、最小闭环、持续验收”的方式逐步落地。

## 2. 里程碑（建议8周）

### 阶段A（第1-2周）：工程基建与骨架
1. 完成Monorepo目录初始化（frontend/backend/contracts/infra/docs）
2. 拉起本地依赖（PostgreSQL 15/Redis/Temporal）
3. 建立前后端最小可运行服务（health check）
4. 建立CI基础门禁（lint、单测、构建）
5. 建立中英语言资源骨架（zh-CN/en-US）

交付物：
1. 可运行仓库骨架
2. 首个OpenAPI文件
3. 后端首个Flyway脚本
4. 首批双语词条与状态映射表

### 阶段B（第3-4周）：核心链路闭环1
1. Workspace管理接口与页面（创建、列表、详情）
2. 需求结构化接口（同步假数据可先行）
3. 前后端契约联调（SDK自动生成）
4. 审计日志最小落库
5. 接口多语言消息返回（Accept-Language）

交付物：
1. Workspace页面可用
2. 需求结构化链路可跑通
3. OpenAPI与实现一致
4. 中英界面可切换

### 阶段C（第5-6周）：核心链路闭环2
1. 任务拆解与估时接口
2. 任务系统同步适配器（先接一种，如Jira）
3. 风险看板接口与页面
4. 周报自动生成草稿

交付物：
1. “需求 -> 任务”闭环
2. 风险看板首版
3. 周报导出接口

### 阶段D（第7-8周）：试点与稳定化
1. 试点团队灰度使用
2. 性能与稳定性优化
3. 权限与审计补齐
4. MVP验收与复盘

交付物：
1. MVP验收报告
2. 版本发布说明
3. 下一阶段Backlog

## 3. 前后端协作机制

1. 每个功能先出OpenAPI草案，再并行开发
2. 前端先接Mock或SDK桩，减少等待
3. 后端接口完成后，前端只做契约内联调
4. 每周固定一次“契约审查会”
5. 每天同步阻塞项与风险项

## 4. 任务分工（建议）

1. 后端主线：
- 领域模型与数据库设计
- MyBatis-Plus Mapper与Service实现
- 工作流与工具适配器
- 鉴权、审计、错误码统一

2. 前端主线：
- 信息架构与页面框架
- SDK接入与请求封装
- Chat、看板、文档中心页面
- 异常态与空态体验

3. 联合任务：
- OpenAPI维护
- E2E关键场景用例
- 发布与回滚预案

## 5. Definition of Done（DoD）

每个需求完成必须满足：
1. OpenAPI更新并评审通过
2. 后端单测通过、前端页面可验收
3. 审计日志可追溯
4. 文档更新到 `docs/`
5. 关键链路有基础自动化测试

## 6. 首批可立即开工任务（T0）

1. 初始化 `frontend/web-console` 项目（Vite + TS）
2. 初始化 `backend` Maven多模块工程
3. 新建 `contracts/openapi/base.yaml`
4. 新建 `infra/docker-compose/local.yml`
5. 新建数据库 `V1__init.sql`（Flyway）
6. 打通第一个端到端接口：`GET /api/v1/health`

## 7. 风险与控制

1. 契约漂移风险：强制以OpenAPI为准并自动校验
2. 进度偏差风险：每周回顾里程碑，拆小任务
3. 质量风险：首批就引入测试门禁，不后补
4. 权限风险：早期就接入基础鉴权和审计

## 8. 协作约定

1. 所有方案变更先更新 `docs/` 再改代码
2. 重大技术决策新增ADR文档
3. 文档和代码同PR提交，避免知识滞后
